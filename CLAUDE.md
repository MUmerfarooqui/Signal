# Signal — Project Guide

Signal is an AI-native product intelligence system that continuously ingests data from a company's tools (support, Slack, analytics, sales calls, engineering), reasons across all of it, and delivers a weekly Intelligence Brief — a prioritized, evidence-backed set of product insights — without anyone having to ask for it.

## What We're Building

The core loop:
1. **Ingest** — pull data from connected tools (Zendesk, Linear, Slack, Amplitude, Gong) on a schedule
2. **Reason** — use Claude to identify cross-source patterns and prioritize insights by impact
3. **Deliver** — push a structured Intelligence Brief to the PM via email/Slack, weekly

The secondary surface is a natural language Q&A interface over the ingested corpus (V2).

## Core Principles (non-negotiable)

1. **Every insight must cite sources.** No unsourced claims. Every bullet in a brief links to the specific tickets, transcripts, or analytics events that generated it.
2. **Push-first.** Signal delivers value to the user; users don't pull it. Brief goes via email or Slack. The dashboard is secondary.
3. **Fewer, higher-confidence insights over comprehensive but weak ones.** In early builds, surface 3–5 strong insights rather than 10 uncertain ones.
4. **No manual input required.** Integrations are OAuth-based and sync automatically. Users should never need to export/upload data.
5. **Express uncertainty.** Claude prompts should surface confidence levels. Do not project false confidence.

## MVP Scope (Phase 1)

Two integrations only: **Zendesk + Linear**

Features:
- OAuth connection for Zendesk and Linear
- Scheduled ingestion of tickets and issues
- Weekly Intelligence Brief generated by Claude
- Evidence drill-down (click any insight → see source documents)
- Basic insight feed (live dashboard between briefs)
- Email delivery of brief (Resend)

Not in MVP: Slack, Amplitude, Gong, Q&A interface, trend tracking, roadmap push.

## AI Strategy

Three distinct layers — don't conflate them:

- **Embeddings** — use OpenAI `text-embedding-3-small` to vectorize every RawEvent on ingest. Purpose-built, ~20x cheaper per token than an LLM, no reasoning needed here.
- **Clustering** — HDBSCAN or k-means over pgvector. Pure math, zero LLM calls. Groups semantically similar events into themes per source.
- **Reasoning** — Claude Sonnet only for the two steps that genuinely need intelligence:
  1. Cross-source correlation: given cluster summaries from Zendesk + Linear, identify patterns that span sources
  2. Brief generation: synthesize the top correlated patterns into plain-language insights with citations
- **Q&A** (V2) — RAG over the ingested corpus, Claude Sonnet for answer generation

Always use prompt caching when calling Claude: cache the cluster summaries / data context as the prefix, vary only the reasoning instructions.

## Working Together

- We're pre-build. Start with backend scaffolding, then connectors, then reasoning layer, then frontend.
- When in doubt, ship the minimal version that proves the core value (the brief). Don't build the Q&A interface until the brief is working well.
- The product research doc is the source of truth for what Signal is. Check it before making product decisions.
- Open questions (pricing model, minimum data threshold, brief cadence) are documented in the research doc Section 9 — surface them when they become build-relevant.
